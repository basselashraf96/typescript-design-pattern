The SOLID principles help developers build scalable, maintainable, and flexible software using **Object-Oriented Design**.

SRP states that **a class should have only one reason to change**.

Analogy: A restaurant has separate staff for cooking, serving, and cleaning. Each has a single responsibility.


class BlogPost {
    constructor(public title: string, public content: string) {}

    savePost(): void {
        console.log("Post saved to database");
    }
}

// ‚ùå Wrong: Mixing responsibilities (saving + displaying)
class BlogPostDisplay {
    constructor(private post: BlogPost) {}

    display(): string {
        return `
${this.post.title}
${this.post.content}
`;
    }
}

OCP states that **a class should be open for extension but closed for modification**.

Analogy: You can install new apps on your phone **without modifying the OS**.


interface DiscountStrategy {
    getDiscount(): number;
}

class RegularDiscount implements DiscountStrategy {
    getDiscount(): number {
        return 10;
    }
}

class PremiumDiscount implements DiscountStrategy {
    getDiscount(): number {
        return 20;
    }
}

LSP states that **a subclass should be substitutable for its base class** without breaking the system.

Analogy: A universal remote should work for all TV brands, without modifying its core design.


class Bird {
    fly(): void {
        console.log("Flying...");
    }
}

class Penguin extends Bird {
    // ‚ùå Wrong: Penguins can't fly, breaking LSP
    fly(): void {
        throw new Error("Penguins can't fly!");
    }
}

ISP states that **a class should not be forced to implement interfaces it does not use**.

Analogy: A coffee machine should not require a function for making tea if it only makes coffee.


interface Printer {
    print(): void;
}

interface Scanner {
    scan(): void;
}

class MultiFunctionPrinter implements Printer, Scanner {
    print(): void {
        console.log("Printing...");
    }
    
    scan(): void {
        console.log("Scanning...");
    }
}

// ‚úÖ Good: Separate interfaces for separate concerns.

DIP states that **high-level modules should not depend on low-level modules. Instead, both should depend on abstractions.**

Analogy: A TV should work with any brand of remote control, not just a specific one.


interface PaymentProcessor {
    processPayment(amount: number): void;
}

class CreditCardProcessor implements PaymentProcessor {
    processPayment(amount: number): void {
        console.log(`Processing Credit Card Payment of $${amount}`);
    }
}

// ‚úÖ Good: High-level module depends on an abstraction, not a concrete class.
class Checkout {
    constructor(private paymentProcessor: PaymentProcessor) {}

    completePurchase(amount: number): void {
        this.paymentProcessor.processPayment(amount);
    }
}
